// Game Flow
🎲

Player creates game

Calls start_game(guess, stake) — guess is 0 (heads) or 1 (tails). Stake must be within house-defined min/max bounds.

🔏

House signs the game ID

House signs game_id using its BLS12-381 private key. Signature is the source of randomness — cannot be predicted before the game is created.

⚖️

Settlement

Contract calls finish_game(game_id, bls_sig). Signature is verified on-chain via bls12381::bls12381_min_pk_verify. Blake2b256 hash determines heads or tails.

💰

Payout

Winner gets double stake minus fee. Fee is taken in basis points (base_fee_in_bp) and accumulates in the house's fee balance, withdrawable by house admin.

// Why BLS Randomness?

Verifiable

The BLS signature is verified fully on-chain. Anyone can re-derive the result from the signature and game ID.

Unpredictable

The house doesn't know which game IDs will be created in advance. The signature can't be pre-computed to cheat.

No oracle dependency

No Chainlink VRF. No external randomness beacon. BLS12-381 is a native Sui framework primitive.

On-chain verification

bls12381::bls12381_min_pk_verify(sig, pk, msg) verifies the house's signature.
blake2b256(sig) hashes it to a single byte — even = heads, odd = tails.

// House Module

HouseData

Holds house balance, fee accumulator, BLS public key, and configurable min/max stake limits. Owned by the house address.

HouseCap

Capability object that gates admin operations — withdraw fees, update stake limits, top up house balance.

Game object

Represents a live game: player address, guess, staked balance, fee amount. Created per flip, destroyed on settlement.
